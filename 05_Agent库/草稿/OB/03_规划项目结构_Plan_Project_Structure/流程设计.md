## 一、为什么是三条独立能力线，不是一条 L3 流程

PTA 的实操经验（SOP v1.2 第2步提醒）：不是所有 Agent 都只有一条线性流程。OB 更明显——巡检、检索、概念笔记提炼三者触发时机、输入输出、依赖关系完全不同，硬画成一条流程图只会互相干扰。三者共享同一套 `tools/`（vault 读取、MCP 调用封装）和 `memory/workspace.py`（状态持久化），但各自独立触发。

```
┌─────────────────────────────────────────────────────────────┐
│                      OB Agent（共享 tools + memory）           │
└─────────────────────────────────────────────────────────────┘
     │                        │                        │
     ▼                        ▼                        ▼
┌─────────┐            ┌─────────────┐          ┌───────────────┐
│ ① 巡检   │            │ ② 检索服务   │          │ ③ 概念笔记提炼  │
│ (定时+手动)│           │ (按需调用)   │          │ (文档变更触发)  │
└─────────┘            └─────────────┘          └───────────────┘
```

## 二、① 巡检能力线

### 2.1 流程

```
┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐
│ symlink   │→│ MCP配置    │→│ MCP Server │→│ F文件      │→│ vault统计  │→│ 同步完整性 │
│ 完整性检查 │  │ 一致性检查 │  │ 连通性检查 │  │ 可读性检查 │  │           │  │ 抽查      │
└───────────┘  └───────────┘  └───────────┘  └───────────┘  └───────────┘  └───────────┘
                                                                                    ↓
                                                                          健康报告.md + 仪表盘
```

### 2.2 输入输出

| 步骤 | 输入 | 输出 | 对应 skill/tool（迁移后） |
|---|---|---|---|
| symlink完整性 | vault根目录 | 每个symlink的目标+是否断链 | `skills/vault_sync_health.py` |
| MCP配置一致性 | 3个客户端mcp.json路径 | 每个终端配置是否指向正确server路径 | 同上，可选 `--auto-fix` |
| MCP Server连通性 | vault路径 | buildIndex能否成功+笔记数/标签数 | 同上，调用 `tools/mcp_bridge.py` |
| F文件可读性 | F1/F2/F3路径清单 | 每份文件是否存在+大小 | 同上 |
| vault统计 | vault路径 | md文件数/目录数 | 同上 |
| 同步完整性抽查 | vault路径 | 随机抽3个文件核对MCP索引是否命中 | 同上 |

沿用现状：定时（launchd，`10_部署与运行_Deploy_and_Run/com.jasper.ob-sync-agent.plist`）+ 手动均可触发，输出健康报告到指定路径。

## 三、② 检索服务能力线

### 3.1 流程

```
┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────────┐
│ 业务Agent  │ → │ 判断查询   │ → │ 混合检索   │ → │ 背景上下文包│
│ 发起请求   │    │ 范围/项目  │    │(关键词+图谱 │    │ 返回给调用方│
│           │    │           │    │ +向量RRF)  │    │           │
└───────────┘    └───────────┘    └───────────┘    └───────────┘
```

### 3.2 输入输出

| 步骤 | 输入 | 输出 | 对应 skill/tool |
|---|---|---|---|
| 查询范围判断 | 调用方项目名/主题 | 限定检索的vault子目录（三项目之一，或跨项目） | `skills/knowledge_retrieval.py` |
| 混合检索 | 查询文本+范围 | RRF融合后的原子列表 | 底层调用 `obsidian-mcp-server` 的 `hybrid_search`（本轮已校准5个bug） |
| 背景上下文包 | 检索结果 | 结构化 dict（原子内容+来源+关系），不是原始文档全文 | `skills/knowledge_retrieval.py` |

### 3.3 服务接口形式（2026-07-15 Jasper 拍板）

暴露独立服务接口，PTA 走薄客户端模式——PTA 只知道"什么时候该问、问完怎么用"，不直接耦合 MCP 协议细节。具体形式待 06 阶段设计（候选：OB 提供一个 Python 函数/CLI，PTA 的 `tools/` 里包一层薄封装调用它；而不是 PTA 直接 import `obsidian-mcp-server` 的 Node.js 代码）。

## 四、③ 概念笔记提炼能力线（全新，本次骨架搭建后优先级最高）

### 4.1 流程（草案，细节待06阶段详细设计）

```
┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────────┐
│ 项目文档   │ → │ LLM提炼   │ → │ 关系挂载   │ → │ 写回vault  │
│ 新增/变更   │    │ 知识原子   │    │(wikilink)  │    │(对应项目目录)│
└───────────┘    └───────────┘    └───────────┘    └───────────┘
```

### 4.2 输入输出

| 步骤 | 输入 | 输出 | 对应 skill/tool（待开发） |
|---|---|---|---|
| 文档变更检测 | 三个项目目录 | 新增/变更文档清单 | 复用 PTA 已验证的 `tools/file_diff.py` 思路（sha256 diff），OB 自己实现一份 |
| LLM提炼知识原子 | 单份文档+已有原子索引（避免重复造原子） | 结构化知识原子（frontmatter+摘要+建议关系） | `skills/concept_note_extraction.py` + `prompts/concept_note_extraction_system.md` |
| 关系挂载 | 新原子+现有原子库 | wikilink 关系（`[[原子名]]`） | 同上 |
| 写回vault | 新原子 | 落盘到对应项目目录下的 .md 文件 | 需要**写入能力**——现有 `obsidian-mcp-server` 的 MCP 连接是只读的，这是本能力线要新补的技术缺口 |

### 4.3 已知的关键设计缺口（不在本文档解决，列为06阶段的详细设计任务）

- 知识原子的粒度定义：一份文档拆几个原子？按什么切（现有 `chunkNote` 是按标题+长度机械切分，"概念笔记"应该是按语义/概念切，两者不是一回事）
- 去重/更新策略：同一份源文档被再次分析时，怎么判断"这个原子该更新还是该新增"
- 写入侧的实现路径：是扩展 `obsidian-mcp-server` 加一个写入 MCP 工具，还是 OB Agent 自己直接写文件（跳过 MCP）
