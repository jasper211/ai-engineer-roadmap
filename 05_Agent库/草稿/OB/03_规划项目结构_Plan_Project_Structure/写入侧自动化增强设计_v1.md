# 写入侧自动化增强设计 v1

## 一、问题重新定位

三大主Agent体系架构7.3节写"现有原子全部靠人工跑离线批处理脚本
（`migrate_full_vault.py`）一次性灌入，不是任何Agent产出文件后自动触发的
实时写入"——但实际审查这个仓库的代码后发现，情况比这句话描述的更细致：

1. **批量+增量提炼编排（`batch_concept_extraction.py`/`agent.py --extract-project`）
   已经写好了**，支持增量检测（`file_diff.py`，sha256对比）、过时原子标记
   （源文档变更/删除后旧原子标"待复核"，不静默变孤儿）、语义去重
   （`atom_embeddings.py`）——这不是"从零设计"的问题。
2. **定时触发的launchd模板也已经写好**（`com.jasper.ob-daily-extract.plist`，
   每天凌晨2点跑EA+Jasper两个项目的增量提炼→同步iCloud→git提交推送），只是
   **README明确记录"不自动安装，需填真实API key"——模板存在，但从未激活**。
3. 真正的技术缺口是三处具体的、此前没人接手的活：
   - **巡检线的plist指向过时路径**（指向已废弃的独立脚本，不是新的
     `agent.py --sync-check`入口）
   - **`write_atom()`只写5个基础字段**（type/concept_type/project/source/
     extracted_at），从不写authority_layer/confidence/decision_status/
     entity_type/entity_ref——这是Jasper AI协同经验引擎418个原子schema
     "贫瘠"的直接原因，也是为什么就算激活daily-extract，产出的还是跟EA项目
     不一致的schema
   - **聚类（entity_hub生成）完全不在这条自动化链路里**，只存在于已经不在
     这个仓库里的一次性脚本`migrate_full_vault.py`

三大主Agent体系架构的判断（"完全没有技术方案"）更多是指"聚类+治理评分"这
两环，不是"提炼本身"——提炼这一环其实已经相当成熟。这次的设计+实现聚焦
这三个具体缺口，不是重新发明整条链路。

## 二、confidence 缺失的根因

`prompts/concept_note_extraction_system.md`（提炼用的LLM系统提示词）此前
只要求输出title/concept_type/summary/related_concepts四个字段，**从没让
LLM判断confidence**——这不是"忘了跑评分"，是设计上就没有这一步。vault里
EA项目原子confidence字段99%以上是UNSTATED占位符，是这个事实的直接后果，
不是"评分逻辑跑了但结果不理想"。

同理，decision_status/entity_type/entity_ref也不该指望靠提炼这一步就填上
真实值——decision_status本质是"治理评审是否通过"的工作流状态，不该在原子
刚创建、还没被任何人/AI看过的时候就自称"已确认"；entity_type/entity_ref
依赖聚类，聚类需要看到"这批原子之间的关系"，单份文档提炼时看不到这个全局
视图，不该也不能在提炼这一步决定。

## 三、本次实现的四处改动

### 3.1 confidence 改为LLM在提炼时一并输出（低成本方案）

修改 `prompts/concept_note_extraction_system.md`（含表格版
`table_extraction_system.md`）：每个原子除了原有字段，新增
`confidence`（HIGH/MEDIUM/LOW）+ `confidence_reason`（一句话理由），判断
标准是"表述的确定性"（原文是肯定语气/带保留语气/纯推断），不是"你相不相信
这是对的"。这个字段可以在同一次LLM调用里一并产出，不需要额外的评分调用、
不增加API成本。

**这只解决"confidence不再是空白"，不解决"confidence判断得准不准"**——
这套标准需要真实数据验证，如果后续发现LLM打分系统性偏高/偏低，需要专门
校准，不是这次改完就一劳永逸。

### 3.2 authority_layer 改为确定性派生（不依赖LLM）

新增 `tools/project_filters.derive_authority_layer(project_name, source_path)`：
EA项目按`EA_LAYER_PRIORITY`各分层目录名，映射到对应authority_layer值
（00_治理与元模型→00_治理，03_发布成果-交付物→03_已锁定，08_任务与跟进→
08_任务跟进，01_原始材料-外部导入→01_原始，02_过程成果-工作产出/*→
02_草稿）——纯字符串前缀匹配，这些映射关系本来就是EA项目自己的治理分层
定义，只是此前没人把它接到写入逻辑上。

Rw权益项目/Jasper AI协同经验引擎目前没有同等细致的分层方案（跟
`project_filters.py`原有注释一致的已知现状），暂时统一给`02_草稿`这个
中性默认值——**这是临时简化，不是最终方案**，等这两个项目的分层定义由
Jasper拍板后需要改成按各自规则派生。

### 3.3 write_atom() 补齐完整schema

`concept_note_extraction.py`的`write_atom()`现在写入authority_layer
（派生）、confidence/confidence_reason（LLM给出）、decision_status
（固定UNSTATED，创建时不该自称已确认）、entity_type（固定"待聚类"）、
entity_ref（固定"（无）"）——跟vault里EA项目原子的字段结构对齐。
`tools/atom_metadata.py`的`FRONTMATTER_FIELDS`同步加入`confidence_reason`，
检索层（`knowledge_retrieval.py`）能读到这个新字段。

**影响范围声明**：这个改动只影响之后新提炼的原子，不会retroactively修改
Jasper AI协同经验引擎现有418个原子（它们仍然是5字段的贫瘠schema）——给
这批存量原子补齐字段是独立的"回填"任务，需要专门写一次性脚本（复用
`derive_authority_layer`本身没问题，但confidence需要重新调LLM判断，这批
存量原子当初提炼时没让LLM给出这个字段），本次没有做，留在"阶段D"里。

### 3.4 新增聚类能力：`skills/cluster_atoms.py` + `agent.py --cluster-project`

新增`AtomClusterer`：扫描某项目下`entity_type: 待聚类`的原子，先尝试按
embedding相似度（阈值0.72，沿用vault里已有枢纽记录的历史阈值）匹配进既有
枢纽（枢纽已达15个原子的规模上限则跳过，不再往里塞），匹配不上的原子之间
做候选聚类（并查集），候选组交给LLM（新提示词
`prompts/atom_cluster_coherence_system.md`）判断是否真的连贯、要不要
拆分，确认连贯的组建成新的entity_hub。

**直接针对已发现的"巨型垃圾桶枢纽"问题设计了两道防线**（"财务流程与凭证"
204个原子那种）：新提示词写死"单枢纽不超过12-15个原子，超过必须拆分"的
硬性规则，模块自己也用`HUB_SIZE_CAP=15`强制这个上限——不管LLM怎么判断，
代码层面就不会往一个枢纽里塞超过15个原子。

命令行入口：
```bash
python3 agent.py --cluster-project "EA流程架构项目" --dry-run   # 免费，只看会形成哪些候选组
python3 agent.py --cluster-project "Jasper AI协同经验引擎" --max-llm-calls 10  # 真实调用，控制成本
```

**已知简化（v1，非最终方案）**：
1. 原子加入既有枢纽只做embedding阈值判断，不二次调LLM确认——假设已验证
   过的枢纽风险较低，如果后续发现真实误判案例需要补上这一步。
2. 候选聚类的连通分量超过`COHERENCE_BATCH_SIZE`(20)时按固定大小切块喂给
   LLM，没做更智能的子分组，真实数据出来后可能要调整切法。
3. 完全没有缓存embedding的原子（没配置embedding API/是embedding功能接入
   之前提炼的存量原子）直接跳过，留在"待聚类"，不退化到其他匹配方式。

**已用合成数据验证过三条路径**（新建枢纽/加入既有枢纽/dry-run不写入），
真实vault数据（尤其EA项目1645个存量待聚类原子）上的表现还没跑过，建议先
对Jasper项目（当前0个既有枢纽，全新聚类）或EA项目的一个小子集做`--dry-run`
+小`--max-llm-calls`验证，再放开跑全量。

## 四、这次没有做、需要Jasper决策后才能做的事

1. **没有安装/激活`com.jasper.ob-daily-extract.plist`**——这个文件本身
   没改动（除了它调用的底层代码现在会产出更完整的schema），是否要
   `launchctl load`让它真正开始每天自动跑，是有真实API成本+自动git推送的
   生产决策，需要Jasper明确拍板，不该由这次改动顺带激活。
2. **已修复`com.jasper.ob-sync-agent.plist`的过时路径**（指向新的
   `agent.py --sync-check`），但同样没有重新`launchctl load`——这个文件
   改了内容，旧的launchd任务如果已经在跑，用的还是加载时的旧版本，需要
   重新加载才会生效。
3. **没有对Jasper AI协同经验引擎现有418个原子做schema回填**——需要专门
   写一次性脚本，且confidence字段需要重新真实调用LLM（会产生费用），
   建议跟阶段D的其他工作一起排期。
4. **聚类脚本没有对真实vault数据跑过**，只在合成测试数据上验证过逻辑
   正确性，建议先`--dry-run`+小样本验证。
