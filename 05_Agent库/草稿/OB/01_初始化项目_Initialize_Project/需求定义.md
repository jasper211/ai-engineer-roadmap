## 一、这个 Agent 解决什么问题？

OB（Obsidian 知识库）Agent 是全部业务 Agent（PTA/VNW/AIT）+ 全部 AI 工具终端（Claude Desktop/Qoder/Kimi Code）共享的**知识库基础设施**，解决三类问题：

1. **知识库本身健不健康**——符号链接有没有断、MCP 配置是否一致、MCP Server 能不能正常建索引、多终端能不能真的连上（继承自现有 `ob_sync_agent.py`，已有实际代码验证过）。
2. **业务 Agent 执行任务前，怎么低成本拿到项目背景**——PTA 现有五个扩展能力都缺"先理解项目背景/文档逻辑关系再做任务抽取"这一层（`--rule-scan` 在真实叙述性文档上噪音很大就是这层缺失的直接后果，见三大主Agent体系架构 3.6 节），这层不该由 PTA 自己重新造，应该由 OB 提供检索服务。
3. **知识库里的内容本身够不够"干净"、有没有结构**——2026-07-15 核实发现，当前 vault 里真正专门产出的概念笔记只有 5 篇（导航/索引类），其余 2000+ 篇是 symlink 直接镜像的 Rw/EA 项目原始文档全文，不是经过提炼的知识原子。继续在这个语料基础上优化检索算法（关键词/向量/图谱），本质是"精确地检索一堆噪音"——**概念笔记提炼**因此被确认为 OB 自己的第一个核心能力，不是外部前置条件。

**2026-07-15 Jasper 决策**：vault 彻底重置——删除现有 symlink 镜像，只保留围绕三个项目（Rw权益项目 / EA流程架构项目 / Jasper AI协同经验引擎本身，含 PTA/VNW/AIT/方法论转正Agent 等 Agent 建设过程本身的经验知识）的概念笔记/知识原子；手机端 Obsidian 刚下载、无历史内容包袱，不构成阻塞。三个项目**合并为一个 vault**、内部按项目分三个顶层目录组织，不拆成三个独立 vault——理由：Obsidian 的图谱能力（wikilink/backlink 遍历）依赖同一个 vault 才能跨项目关联（PTA 的踩坑经验会引用 EA 的价值节点方法论、也用过 Rw 的真实数据验证过），且现有 `obsidian-mcp-server` 的 `buildIndex(vaultPath)` 是单 vault 路径架构，合并不需要额外改造。

## 二、谁用这个 Agent？

| 使用者 | 用途 |
|---|---|
| **Jasper** | 查看巡检健康报告；定义/审核概念笔记提炼的质量标准（原子粒度、关系是否准确）；触发 vault 重置/重建 |
| **PTA（未来 VNW/AIT）** | 检索服务的消费方——执行 `--discover`/`--intel`/`--rule-scan` 等分析前，先问 OB 拿项目背景上下文包，注入 LLM 提示词或用于判断是否跳过某文档；不直接读写 vault 文件本身 |
| **多 AI 终端**（Claude Desktop/Qoder/Kimi Code） | 通过各自的 `obsidian-knowledge` MCP 连接，直接调用检索工具（`search_notes`/`hybrid_search`/`get_note_graph` 等），不经过 PTA |

## 三、成功标准是什么？（按三条能力线分别定义）

| 能力线 | 成功标准 |
|---|---|
| ① 巡检 | 沿用现有标准：symlink/MCP配置/MCP Server连通性/F文件/vault统计/同步完整性抽查，全部通过时报告"🟢 全部正常"；异常时明确指出修复建议 |
| ② 检索服务 | 给定一个项目+主题，5秒内返回相关的概念笔记原子（不是原始文档全文）+ 关系上下文；PTA 接入后，`--rule-scan` 真实叙述性文档的误抽取率相比现状（290条"任务"里大量噪音）有可衡量的下降 |
| ③ 概念笔记提炼 | 针对三个项目中的任一份新增/变更文档，能产出结构化的知识原子（含 frontmatter：来源文档/项目归属/概念摘要 + 与已有原子的 wikilink 关系），不是简单的整篇文档摘要 |

## 四、三条能力线边界（替代单一 L3 流程——OB 参照 PTA"两条独立能力线"的处理方式，不强行拼成一条线）

| | ① 巡检 | ② 检索服务 | ③ 概念笔记提炼 |
|---|---|---|---|
| **触发方式** | 定时（launchd）+ 手动 | 被业务Agent/AI终端按需调用 | 项目文档新增/变更时触发，或手动批量重建 |
| **现状** | 已有实际代码 `ob_sync_agent.py`（648行，6项检查），待迁移进 skills/tools | 底层 `obsidian-mcp-server` 已完成关键词+图谱+向量混合检索校准（本轮修复5个真实bug），待补一层对 PTA 暴露的薄客户端接口 | 全新能力，未有任何代码，是本次骨架搭建后最优先要设计实现的部分 |
| **依赖关系** | 独立，不依赖②③ | 依赖③产出的概念笔记（当前语料是 symlink 原始文档，重置后依赖③重新产出） | 依赖 vault 重置完成（symlink 移除）后才能从"干净的起点"开始产出 |
| **对应 skill（草案）** | `skills/vault_sync_health.py` | `skills/knowledge_retrieval.py`（薄客户端封装，PTA 端调用） | `skills/concept_note_extraction.py`（新，需配套 `prompts/concept_note_extraction_system.md`） |
