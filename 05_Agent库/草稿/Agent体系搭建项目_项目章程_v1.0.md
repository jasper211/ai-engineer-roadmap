## 一、项目章程

### 1.1 为什么要独立成项目

Agent体系目前的正式身份是「AI工程能力整改项目」里"维度2·Agent/Skills设计"这一行——看板上显示85%，但这个百分比是PTA（v2.8.2，已验证很深）掩护出来的整体观感，实际上五个Agent里进度差异很大（PTA成熟、OB接近成熟、VNW/AIT早期、方法论转正Agent未启动）。这种"一行进度条代表五个独立复杂度不同的系统"的记法，已经不足以支撑真实的规划和沟通，需要五个Agent各自的进展、依赖关系、风险，在一个独立的项目空间里被看见，而不是被压缩成整改看板上的一个数字。

### 1.2 项目目标

把此前靠人工经验推动的「价值节点 → 规则空白 → 访谈 → SOP」闭环，以及配套的 AI 协同方法论，转化为一套可复用、可验证、能对外展示的 Agent 体系。**（v1.1更新）**核心是 PTA/VNW/AIT/方法论转正Agent 四个Agent协同工作，覆盖"通用任务执行、业务规则挖掘（发现引擎）、SOP的AI化延伸（三轨道拆解）、方法论沉淀"四类职能；OB作为第五方提供"知识背景支撑"，但OB自身的建设由另一个独立项目负责，本项目只对接它已就绪的读取服务。VNW与AIT是同一条流水线的前后段（VNW产出SOP即触发AIT介入），不是相互独立、可以分别排期的两个模块。

### 1.3 项目范围

**在范围内：**
- PTA / VNW / AIT / 方法论转正Agent 四个Agent的规划、构建、集成、验证、归档
- **（v1.1更新）VNW→AIT的强耦合流水线设计**：节点锚点模型、规则级溯源模型（VNW侧）+ SOP三轨道拆解、协同新规则判定机制（AIT侧）——两者是同一条流水线的前后段，不是独立模块，规划时要一并考虑
- 支撑这些Agent的共享方法论文档（Agent搭建SOP、验证标准清单、体系架构文档）的维护
- Agent之间的协同关系设计（复用关系、调用关系、client-service接口），**包括业务Agent对OB既有读取服务（`get_context()`）的调用侧接线**——接口设计和调用代码在本项目范围内，OB服务端本身不在
- 定期向「AI工程能力整改项目」维度2同步进展（该项目仍是上级项目，本项目是其内容来源，不是替代）

**不在范围内：**
- EA流程架构项目本身的业务分析工作（规则前端设计、process-db-mcp、价值节点清单的内容本身）——这些是Agent的"目标项目"或消费方，不是本项目要交付的业务成果，本项目只负责"用Agent处理它们"这件事本身
- 「AI工程能力整改项目」里其他10个能力维度（AI Coding工具链、模型选型矩阵、技术雷达、飞书集成、成本优化、内容产出、作品集网站等）
- **（v1.1更新，原属在范围内）OB自身的搭建/优化**——本体化、检索层、写入侧、自巡检机制，架构v1.3已明确划到另一个独立项目，不占用本项目资源；本项目对OB只有"调用侧接线"这一件事在范围内（见上）
- OB vault 里知识内容本身的准确性（属于知识管理范畴，不属于Agent工程范畴）

### 1.4 成功标准

- **（v1.1更新）**PTA/VNW/AIT 三个本项目直接建设的"重活型"Agent各自完成归档确认（Mark/mentor确认，`archive_gate.mark_confirmed = true`）——OB不再是本项目的归档对象
- Agent验证方法论四原则（确定性检查优先/独立数据源交叉核验/异常特征清单优先/验证标准独立文档化）在所有已上线Agent的测试套件上复核通过，不是"测试全过"就默认合格
- **（v1.1更新）**至少一条VNW→AIT的真实端到端案例跑通（拿一个已产出SOP的节点子集，走完节点锚点→SOP→三轨道拆解全程，参照L3-COM手工案例校准），而不是停留在矩阵里"仅COM手工案例"这句判断上（Pipeline矩阵阶段3当前6/12，是全矩阵最低分之一）
- 项目自身的看板/风险清单/决策日志持续更新，进展可以不靠记忆、随时向Mark/mentor说清楚

### 1.5 利益相关方

| 角色 | 关注点 |
|---|---|
| Jasper | 项目Owner + 主要执行者 |
| Mark | 决策方/mentor，关注整体进展是否支撑既定的能力整改叙事，部分Agent的归档需要他确认 |
| 各AI协作终端（Claude/Kimi/Codex/Qoder） | 执行层，需要遵循同一套Agent搭建SOP和验证标准，不各自发明规则 |

### 1.6 与「AI工程能力整改项目」的关系

本项目不是「AI工程能力整改项目」的替代，是它维度2的内容来源和执行细节的展开。两者关系：本项目的进展定期（建议随周五复盘节奏）摘要同步进「能力整改看板.md」维度2一行；反过来本项目不需要重复维护整改项目的其他10个维度。如果未来两个项目的看板出现数字对不上的情况，以本项目的现状基线为准，整改看板做一次校准。

## 三、路线图与优先级排序

**（v1.1更新：改按架构v1.3十-2节的顺序，替换v1.0基于v1.2的旧排序，核心变化是AIT提前、OB移出）**

1. **VNW补齐**（**当前建议节点**）——把Phase1已验证的4个脚本按SOP六步组织成真正的Agent，复用PTA已验证的01-11骨架模板，同时落地节点锚点模型+规则级溯源模型；具体建议先做"基线合并（批次2）"，为下一步AIT MVP准备可用的SOP产出
2. **AIT同步启动（MVP）**——不等VNW全量完成，试点节点已拍板选**L3-COM**（[D-20260721-001](../../02_decisions/D-20260721-001_VNW到AIT首个试点节点选择.md)），先把L3-COM既有的8份手工文档倒推成VNW标准产出格式（节点锚点+规则级溯源SOP），再让AIT基于此做三轨道拆解原型——这是矩阵里6/12分的VNW→AIT断点，也是当前最该优先打通的链路
3. **PTA验证环节改造**——按四原则复核`test_integration.py`，验证"Agent验证方法论"本身可行性的第一个真实案例
4. **方法论转正Agent启动**——在②验证出第一批"协同新规则判定"真实案例、①③都跑通方法论四原则可行之后再建
5. **OB对接**——待另一个项目把OB自巡检/写入侧建完后，PTA/VNW/AIT再各自补上调用侧代码；**本项目当前不主动推进**，只在VNW/AIT产出格式上做好衔接准备（溯源标注等）
