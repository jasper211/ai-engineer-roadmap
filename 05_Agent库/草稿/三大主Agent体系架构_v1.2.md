## 一、设计背景

### 1.1 为什么需要 Agent 体系

Phase 1 完成了多个自动化脚本（信号提取、规则合并、一致性校验），但存在以下问题：

| 问题 | 表现 | Agent 化解决 |
|------|------|-------------|
| 脚本分散 | 每个脚本独立运行，无统一调度 | 主 Agent 统一编排 |
| 触发被动 | 需要用户手动执行 | 子 Agent 主动监控 + 触发 |
| 产出孤立 | 信号基线、规则空白、看板更新各自为政 | 端到端自动同步 |
| 认知断层 | 每次新对话需重新理解上下文 | Agent 配置持久化上下文 |
| **（v1.1新增）方法论各自发明** | PTA/VNW/AIT各自摸索验证标准，没有共享标准，AI工具间调用规范也不统一 | 方法论转正Agent统一沉淀验证标准；Obsidian巡检Agent统一AI工具调用准则 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **L3 端到端** | 业务产出型主Agent（PTA/VNW/AIT）对应一个完整的 L3 流程，子 Agent 对应 L4/L5 活动 |
| **单一职责** | 主 Agent 不跨域，PTA 管项目、VNW 管价值节点、AIT 管转型咨询 |
| **可组合** | 子 Agent 可被多个主 Agent 复用（如 PTA-S04 文档同步器可被 VNW 调用） |
| **渐进实现** | 先实现 VNW（业务价值最直接），再 PTA（通用性最强），最后 AIT（战略价值）——**实际推进顺序已偏离原计划，PTA先跑到v1.2.0，见九、实现路线图** |
| **（v1.1新增）验证独立** | 所有Agent的"验证"环节用Agent完成，不用人工；验证标准统一由方法论转正Agent维护，不各自发明（详见D-20260709-001） |
| **（v1.1新增）面向复用设计** | 部分Agent未来要推向公司使用，不只是个人资产——配置和任务知识库需可插拔（参照PTA v1.2.0的`pta_tasks.json`外置模式），不能硬编码个人工作路径/习惯 |

## 三、主 Agent 1：项目任务协同 Agent (PTA)

### 3.1 定位

通用型项目执行引擎，适用于任何项目的任务全生命周期管理。**（v1.2现状）已迭代到 v2.8.2**：01-11 方法论骨架（`01_初始化项目`~`11_监控与优化`）完整落地，agents/skills/tools/memory/prompts/tests 六个 Python 包各自嵌在对应编号目录下——**这套骨架本身已经作为"以后任何 Agent 搭建都要遵循的标准结构"验证过一遍，是本次 v1.2 更新里唯一一条对其余四个 Agent 直接有参考价值的结论**。核心循环 + 主动巡检 + 五个扩展分析能力全部完成，24 项集成测试全过，草稿态待 Mark 确认归档。

### 3.2 两条能力线（原"L3端到端流程"，v1.2改为按实际架构描述）

PTA 现在是两条独立但共享底层工具的能力线，不再是单一线性流程：

**① 被动执行（Think-Act-Observe 主循环）**——响应一句自然语言指令：
```
意图解析(intent_parsing) → 执行编排(execution_planning) → 进度追踪(progress_tracking)
    → 归档复盘(archive_review，本地写入，无 git 动作，始终执行)
    → [仅显式 --sync 时] 产出同步(doc_sync，唯一含真实 git push 的阶段)
```

**② 主动巡检（daily_sensing，v2.3.0 新增，独立入口 `--daily-scan`）**——不需要用户先发指令，每天定时主动跑：
```
本地文件 sha256 diff（零成本）→ 合并一次 LLM 关系分析/相关性判断
    → 建议任务铸造成 RPT-YYYYMMDD-NN → 安全 merge 进目标项目 pta_tasks.json
    → [仅显式 --notify] 企业微信群机器人推送
```
执行前必须人工确认（`agent.py "执行 RPT-xxx" --execute`），走的是①的执行管线，不做全自动执行。已接入 launchd，在真实 EA 项目上每天 10:00 自动运行、真实推送企业微信。

### 3.6 v1.2新发现：真实数据验证暴露的问题 + 已识别但未填补的能力缺口

**真实数据验证价值实证**：三批迁移完成后，第一次用真实 EA 项目 + 真实 Rw 权益项目（40条真实跟踪项，402个真实文件）做验证，而不是只用手造的合成测试数据——共发现并修复 5 个此前测试没测出来的真实 bug（daily_sensing 指纹去重失效导致重复铸造RPT任务；project_intelligence 的 Rw 后端进度统计算出负数；`blocker != "无"` 精确字符串比较误判真实台账里"无(说明)"格式；Rw 项目探测逻辑不递归查子目录，指向项目真正根目录时失效；同名台账文件在真实项目里有多份拷贝，查找逻辑选错版本）。**结论：合成 fixture 能验证逻辑对不对，但只有真实数据的"脏"（重名文件、BOM、字段里混着说明文字的自然语言写法）才能验证工具在真实世界里到底顶不顶用**——这条经验对 VNW/AIT 未来做类似验证时同样适用。

**已识别但故意未填补的缺口——"背景记忆层"**：`--rule-scan` 在真实叙述性文档（工作日志/决策记录类）上噪音很大（真实数据跑出290条"任务"/552条"风险"，很多是从历史记录误抽取的）。深入讨论后确认根因不是关键词规则不够严，而是 PTA 现有五个扩展能力都没有"先理解项目背景/文档之间的逻辑关系，再做任务抽取"这一层——`--rule-scan`零上下文单文件抽取，`--discover`单文件独立喂LLM无项目背景，`--daily-scan`/`--intel`虽然分别做了"变化关系分析"和"跨文档关联"，但前者背景来自静态人工文件（`pta_focus.md`）、后者是规则层面的浅关联，都不是持续积累、越用越厚的语义记忆。**这层缺口的正确填补方式是接入 Obsidian巡检Agent（OB）已在设计的知识图谱能力，而不是PTA自己重新造一遍**，详见七节v1.2更新 + 九-2节"PTA↔OB接口设计"。
