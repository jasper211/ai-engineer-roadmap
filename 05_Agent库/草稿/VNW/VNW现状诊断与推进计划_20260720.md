## 二、与 PTA / OB 的协同关系

1. **下游依赖尚未成立**：`config.json` 里 VNW 的 `downstream` 写的是 `PTA-S04`，即 VNW 产出的规则空白地图/SOP 底稿本该被 PTA 消费。但这个环节 VNW 还没做到，等于这条依赖线目前是空的——PTA 现在也没有真正等这个输入在用。
2. **OB 的下一步不包含 VNW**：OB README 明确写"下一步：PTA↔OB 真正接线"，VNW 没有被排进这一轮。如果这轮精力都花在 PTA/OB 互联上，VNW 大概率会被继续搁置。
3. **有现成资产可以抄近路**：三个 Agent 有相当一部分底层工具是互相移植的（比如 OB 的 `llm_client.py`/`file_diff.py` 就是从 PTA 移植的）。VNW 要做的「基线合并」在 Phase1 阶段已经有验证过的原型——能力整改看板记录的 P1-04（`ManualBaselineParser` + `BaselineMerger`，228 行，119 条规则全部命中）——大概率可以像 `signal_extractor_legacy.py` 一样直接迁移复用，不用从零写。

## 三、推进计划（建议分批，不建议三线并进）

考虑到 Jasper 目前已经在同步推 PTA 和 OB，VNW 建议只启动一个最小批次，其余排入 backlog 但不同时开工：

### 建议本轮做（批次 2）：基线合并 S04
- 直接复用 Phase1 的 `ManualBaselineParser`/`BaselineMerger` 逻辑，按 VNW 已验证的"迁移已有代码而非重写"模式接入。
- 完成后 VNW 首次具备"自动基线 + 手工基线 → 完整基线"的真实产出，是后面规则空白地图生成的必要输入。
- 工作量参照 S01-S03 的迁移经验，预计和上一批体量接近。

### Backlog（暂不启动，先占位）
- 批次 3：规则空白地图生成（S05/S07）——可参考 P1-01 PoC 已经跑通的 437 行脚本 + HTML 报告生成先例。
- 批次 4：SOP 底稿生成（S06）。
- 批次 5：部署与监控（对应 10/11 步）——参考 PTA 的 launchd 每日巡检、OB 的 sync-agent plist 模式。
- 与 PTA 的下游接线（VNW 产出 → PTA-S04 消费）——等批次 3 有真实规则空白地图产出后再排期，目前排太早。
