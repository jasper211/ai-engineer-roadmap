## 一、四条原则（复述，详细论证见D-20260709-001）

1. **确定性检查优先**：能用脚本/规则判定对错的，不用LLM主观判断
2. **独立数据源交叉核验**：评估依据不能只看生成者自己产出的东西，要有独立于生成过程之外的锚点
3. **异常特征清单优先于全量通读**：维护具体、可检测的异常模式清单，不逐字读全部输出找感觉
4. **验证标准独立文档化、跨Agent复用**：本文档就是第4条原则的产出物

## 二、PTA复核实例（第一个真实案例，2026-07-09）

### 2.1 复核前的状态

`test_pta_integration.sh`有8个测试用例，全部通过（exit 0），但复核发现三类问题，都不是"测试失败"，是"测试本身的设计有漏洞，可能让真实的失败被漏判"：

| 问题 | 具体表现 | 对应原则 | 严重度 |
|---|---|---|---|
| `set -e`没配`pipefail` | `python3 ... \| tail -N`模式下，若python脚本崩溃，管道退出码是tail的（几乎总是0），set -e拦不住 | 原则1（确定性检查本身不确定） | 中——本次审查发现每处这个模式后面恰好都有独立的文件/目录存在性检查兜底，是巧合不是设计保证 |
| 文档描述的安全机制与代码不符 | 架构文档写"真实git push需要显式`--execute`确认"，但S04代码根本没有`--execute`参数，真实机制是`--dry-run`不传就是真实执行（默认危险，不是默认安全） | 原则2（独立核验，不能只信文档说法） | 高——是文档层面的错误认知，可能误导后续开发者以为"不加--execute就安全" |
| 只测"应该发生"，没测"不应该发生" | 8个用例全是正向验证（S01能不能解析、S04 dry-run模式对不对），没有一条测"S04默认（不传--dry-run）时是否真的会执行"这种反向/边界用例 | 原则2+4 | 高——安全相关的边界行为最容易被"正向测试全过"掩盖 |

### 2.2 修复内容

1. `test_pta_integration.sh`：加`set -eo pipefail`；Test 2/Test 5改成`if pipeline; then`直接判断（避免pipefail让set -e在到达检查语句前提前终止）；新增Test 9，在隔离无remote的临时仓库里验证S04默认行为，安全（push必然本地失败，不碰真实远端）
2. `三大主Agent体系架构_v1.1.md`：修正PTA-S04那行的"需要--execute确认"错误描述

### 2.3 本次审查暂未覆盖的部分（如实记录，不假装做全了）

- **原则3（异常特征清单）本次没有真正建立**——目前清单里的检查项都是"具体某个值对不对"，还没有沉淀出跨Agent通用的"异常模式"（比如EA项目那边验证过的"模板化复制粘贴""数值区间和结论对不上"这类特征，在PTA这种代码类Agent上该怎么对应，还没想清楚）
- **内容层面的校验仍然薄弱**：S03/S05等几处目前还是"文件存在就算过"，没有校验产出内容本身对不对（比如执行记录里的任务ID是否真的对得上、进度报告的具体字段是否合理）——这次没有动，因为需要先确认预期值该是什么，不想凭空编断言

## 三、给VNW/AIT/OB巡检的检查清单（基于PTA经验总结，开发到第5步时逐条过）

- [ ] 脚本里凡是`cmd | tail/grep/head`这类管道，检查是否开了`pipefail`；如果用`grep -q`且后面要跟着查`$?`，改成`if pipeline; then`直接判断，不要"跑完再查$?"
- [ ] 凡是文档/README里写了"安全机制是XX"，回头对着实际代码核实一遍参数名和默认值方向，不要假设文档写的就是代码做的
- [ ] 每个"应该发生"的正向测试，想一下有没有对应的"不应该发生"的反向测试没写（尤其是涉及外部副作用的操作，如写文件、git操作、发送消息）
- [ ] 涉及真实外部副作用（git push、发消息、写生产文件）的验证用例，必须设计成隔离环境（临时目录/无remote仓库/mock），不能直接对着真实环境跑测试
- [ ] 目前还没有的：本Agent有没有"生成器输出彼此高度雷同"这类模板化异常的检测——如果有批量生成场景（比如VNW批量生成规则空白），参考EA项目"模板化复制粘贴"发现的检测思路
