## 一、这个 Agent 解决什么问题？

### 1.1 现状痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **任务理解偏差** | AI 协同终端误解用户意图（如把"前端草稿"当核心文档） | 返工、时间浪费 |
| **执行碎片化** | 每个任务独立运行，无统一调度 | 重复劳动、状态丢失 |
| **文档同步滞后** | 任务完成后手动更新看板、执行记录、Git | 容易遗漏、版本不一致 |
| **认知断层** | 每次新对话需重新理解上下文 | 效率低下、理解偏差 |
| **进度不透明** | 用户不清楚任务执行到哪一步 | 焦虑、频繁询问 |

### 1.2 核心问题

> **如何让 AI 协同终端像「资深项目经理」一样，准确理解任务、自主执行、实时同步、自动归档？**

## 二、谁用这个 Agent？

| 角色 | 使用方式 | 期望价值 |
|------|---------|---------|
| **Jasper（用户）** | 自然语言指令触发 | 减少沟通成本，一次指令即可推进 |
| **AI 协同终端** | 读取配置、执行子 Agent | 有明确边界，减少理解偏差 |
| **Mark（审核者）** | 查看执行记录和产出 | 可追溯、可审计、可复现 |
| **未来新成员** | 阅读 Agent 文档快速上手 | 降低协作门槛，标准化流程 |

## 三、成功标准

### 3.1 量化指标

| 指标 | 现状 | 目标 | 测量方式 |
|------|------|------|---------|
| 任务理解准确率 | ~70%（需多次纠正） | >90% | 用户纠正次数 / 总任务数 |
| 文档同步及时率 | ~50%（经常遗漏） | >95% | 任务完成后 5 分钟内同步完成 |
| 执行进度透明度 | 低（用户需主动询问） | 高（每步自动汇报） | 用户主动询问次数 |
| 新任务上手时间 | 10-15 分钟（上下文恢复） | <2 分钟 | 从指令到开始执行的时间 |

### 3.2 定性标准

- [ ] 用户说一次指令，AI 能正确理解并执行
- [ ] 任务执行过程中，用户能实时看到进度
- [ ] 任务完成后，所有文档（看板、执行记录、Git）自动更新
- [ ] 新对话开始时，AI 能快速恢复项目上下文

## 四、需求边界

### 4.1 在范围内（Do）

- 理解自然语言任务指令
- 分解任务为可执行步骤
- 调度子 Agent / 脚本执行
- 监控执行进度并汇报
- 任务完成后自动同步文档
- 生成执行记录和复盘

### 4.2 不在范围内（Don't）

- 不替代用户的决策（如技术选型、优先级判断）
- 不执行需要用户密码/密钥的操作（如 sudo、GitHub OAuth）
- 不修改生产环境文件（只读原则）
- 不处理非项目相关的通用任务（如查天气、翻译）

## 六、风险与假设

| 风险 | 影响 | 应对 |
|------|------|------|
| 用户指令过于模糊 | 理解偏差 | 设计澄清机制（PTA-S01 主动提问） |
| 任务执行过程中出错 | 阻塞 | 设计异常处理 + 回退机制 |
| 子 Agent 调用失败 | 流程中断 | 设计重试 + 降级方案 |
| 状态文件损坏 | 记忆丢失 | 定期备份 + 恢复机制 |

| 假设 | 验证方式 |
|------|---------|
| 用户愿意用结构化指令 | 观察实际使用频率 |
| 项目目录结构稳定 | 定期检查路径有效性 |
| Git 仓库可用 | 每次提交前验证 |
