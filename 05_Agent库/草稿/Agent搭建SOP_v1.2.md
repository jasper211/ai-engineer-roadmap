## 一、Agent 是什么？

### 1.1 通俗理解

Agent = **自动化机器人**，它有：
- **大脑**（JSON 配置）：知道自己是干什么的、能调用什么工具
- **感官**（监控/输入）：感知外部变化（文件变更、用户指令）
- **手脚**（子 Agent/脚本）：执行具体任务（解析 Excel、生成报告）
- **记忆**（状态文件）：记住上次做了什么，避免重复

### 1.2 与脚本的区别

| 维度 | 脚本（Script） | Agent |
|------|-------------|-------|
| **触发方式** | 手动执行 | 自动监控 + 触发 |
| **上下文** | 每次从头开始 | 有状态记忆 |
| **协作能力** | 独立运行 | 可调用其他 Agent |
| **配置化** | 硬编码 | JSON 配置驱动 |
| **可复用** | 复制粘贴 | 标准化接口调用 |

### 1.3 Agent 的组成部分

**v1.2更新：以下是原始设想的扁平结构，已被 PTA 实操验证后的 01-11 编号骨架取代——新 Agent 直接照抄 01-11 结构，不要再用这份扁平范例。**

```
{AgentID}/
├── 01_初始化项目_Initialize_Project/         ← 需求定义.md 放这里
├── 02_配置项目_Configure_Project/             ← settings.json（原config.json）+ .env.example + wecom_config等敏感配置模板
├── 03_规划项目结构_Plan_Project_Structure/    ← 流程设计.md
├── 04_定义Agent_Define_Agent/
│   └── agents/agent.py                       ← 主循环入口（原"主编排器"），CLI参数解析+分发
├── 05_集成工具_Integrate_Tools/
│   └── tools/{各技术能力模块}.py              ← 可被多个skill复用的纯技术层（文件diff/LLM调用/git操作等）
├── 06_开发技能_Develop_Skills/
│   └── skills/{各业务能力模块}.py             ← 原"子Agent"，现在是描述性命名的Python模块，不是S01/S02编号脚本
├── 07_接入记忆_Integrate_Memory/
│   └── memory/workspace.py                   ← 状态持久化+专属工作区隔离（跟目标项目物理隔离）
├── 08_设计提示词_Design_Prompts/
│   └── prompts/{skill}_system.md             ← 每个用到LLM的skill，系统提示词独立文件，不内嵌在.py里
├── 09_测试与调试_Test_and_Debug/
│   └── tests/test_integration.py             ← 集成测试唯一入口
├── 10_部署与运行_Deploy_and_Run/              ← launchd plist等定时任务配置模板
├── 11_监控与优化_Monitor_and_Optimize/        ← 严格对应方法论第11步"监控Agent自己"，不要把"帮Agent分析别的项目"的工具也塞进来
├── README.md
├── requirements.txt                           ← 只有真正用到非stdlib依赖时才需要，明确写清楚哪个skill用到、缺失时是否优雅降级
└── _retired_flat_structure/                   ← 旧版本/被迁移替代的脚本存放处，不删除，标注"不再是入口"
```

编号本身只是给人看的顺序标识，实际 Python 包名（agents/skills/tools/memory）不带编号前缀（Python 标识符不能以数字开头）——`sys.path` 要把每个编号目录加进去，而不是 Agent 根目录本身，才能让 `from skills.xxx import` 正常解析。

## 第4步：子 Agent 开发（2-3天）

### 4.1 每个 skill/tool 的标准结构（v1.2更新：取代原S01/S02单文件命名法）

**"子Agent"这个称呼和`{AgentID}-S{序号}_{名称}.py`命名法是早期设想，PTA实操后已经被 skills/tools 模块取代**——不再用编号区分先后，用描述性模块名直接说明这个模块做什么（如 `skills/daily_sensing.py`、`skills/project_intelligence.py`），职责边界更清楚，也不会出现"S03到底是干什么的"这种要翻文档才知道的问题。

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
"""
技能：{一句话描述这个模块做什么}

{如果这个模块是从别的地方合并/迁移过来的，在这里注明来源和迁移时的关键决策——
不是遗留就直接照搬，看下面"五、迁移已有资产"整节}
"""

from pathlib import Path
from typing import Optional

class {DescriptiveName}:
    """{类做什么，不是"处理逻辑"这种空话，说清楚具体职责边界}"""

    def __init__(self, project_root: Path):
        self.project_root = Path(project_root)

    def run(self, previous_state: dict) -> "tuple[dict, dict]":
        """接收状态dict、返回(结果, 更新后的状态dict)——跟memory.workspace的
        读写职责分离：本模块不直接读写状态文件，调用方（agent.py）负责加载/保存。"""
        ...
```

### 4.2 开发 checklist

每个 skill/tool 必须满足：

- [ ] 有清晰的 docstring（功能、跟其他skill的关系边界、如果是迁移来的注明来源）
- [ ] 不直接读写 `memory.workspace` 状态文件——接收状态dict、返回更新后的状态dict，调用方负责持久化
- [ ] 有错误处理（明确的异常类型，不是裸 `except:`）
- [ ] 有返回值（结构化 dict/dataclass，便于上游调用）
- [ ] 涉及LLM调用的，系统提示词放独立的 `prompts/{skill}_system.md` 文件，不内嵌在.py字符串里
- [ ] 涉及外部项目路径的，跟 PTA 自己的工作区/源码所在仓库物理隔离（见 `memory/workspace.py` 的隔离原则）

### 4.3 输出物

- [ ] 所有 skill/tool 的 `.py` 文件
- [ ] 集成测试里对应的测试用例（不要求每个模块单独一个测试文件，`test_integration.py`统一维护）

## 第5步：集成测试（1天）——含Agent验证方法论（v1.2更新新增第5条原则）

### 5.1 测试层级（v1.1更新：端到端测试从"手动验证"改为"Agent验证"）

| 层级 | 测试内容 | 工具 |
|------|---------|------|
| 单元测试 | 每个模块独立运行 | 内嵌在`test_integration.py`里的白盒调用 |
| 集成测试 | 模块串联运行 | `test_integration.py` |
| 端到端测试 | 完整流程 | **Agent验证**（不再是手动验证，见5.2） |
| **（v1.2新增）真实数据验证** | 用真实项目数据跑一遍，不只是合成fixture | 见5.2第5条原则 |

### 5.2 Agent验证方法论·五条原则（v1.2新增第5条，详见D-20260709-001）

⚠️核心原则：验证者必须独立于生成者，不能自己给自己打好评——这不是"找另一个LLM实例读一遍生成结果、主观判断对不对"，那种做法容易和生成者共享同一类偏见，一致性地漏判同一类错误。

1. **确定性检查优先**：能用脚本/规则判定对错的，不用LLM主观判断。这条原则本项目里已经在用——`test_integration.py`就是脚本判定通过/失败，不是找AI去看。任何新Agent的验证设计，先问"这一步能不能写成断言/数值核对/schema校验"，能写就不要交给LLM判断。
2. **独立数据源交叉核验**：评估者的判断依据不能只看生成者自己产出的东西，要有一个独立于生成过程之外的锚点。例：验证文档同步器，不是"AI读一遍commit信息觉得写得好"，是核对"是否真的push成功、文件是否真的发生了变更"这类可以从外部独立核实的事实。
3. **异常特征清单优先于全量通读**：不要求评估者逐字读完全部输出再给主观感想，是维护一份具体、可检测的异常模式清单（例如"数值区间和结论对不上""输出内容明显是模板复制粘贴"），评估者专门扫描这些特征，命中就标红。这份清单需要跟着实际踩过的坑持续补充，不是一次性写完。
4. **验证标准独立文档化、跨Agent复用**：不能每个Agent的验证标准各自发明一套，要维护一份共享的"Agent验证标准清单"（方法论转正Agent建成后由其负责持续维护），新Agent开发到这一步先看这份清单里有没有现成的，不要重新摸索。
5. **（v1.2新增）真实数据验证优先于/补充合成测试**：合成 fixture 只能验证"逻辑对不对"，不能验证"工具在真实世界里到底顶不顶用"——只有真实数据的"脏"才能把假设错误的地方暴露出来。**PTA的真实实证**：三批扩展脚本迁移完成、24项合成测试全过之后，第一次用真实EA项目+真实Rw权益项目数据验证，又抓出5个此前测试完全没测出来的真实bug：
   - 真实字段里混着自然语言说明（`blocker="无(Roy财务口径已确认...)"`，不是干净的`"无"`），精确字符串比较全部误判
   - 真实项目目录结构跟假设不符（数据文件嵌套在子目录里，不在假设的根目录），路径查找逻辑指向"真正的项目根"时直接失效
   - 真实项目里同名文件存在多份拷贝（不同Agent各自的工作区快照），查找逻辑毫无防备地选中了错误的版本
   - 独立LLM调用间的输出存在细微不确定性（同一件事两次调用返回的关联字段顺序不同），基于精确匹配的去重逻辑被打穿，产出重复数据
   - **结论/操作要求**：新Agent走到第5步，必须至少用一份真实项目的真实数据跑一遍全流程（不能只是团队内部虚构的示例数据），且要刻意找"数据比较脏"的真实项目（文件多、格式不统一、历史遗留多）作验证对象，而不是挑一个干净的样板项目应付了事。

## 五、迁移已有资产进Agent架构的方法论（v1.2新增，来自PTA三批扩展脚本迁移的真实经验）

**适用场景**：手上已经有一批验证过、能跑的脚本/工具（不是从0写），要把它们组织进正式的Agent架构里。**VNW接下来要做的事正是这一类**——Phase1已经有4个验证过的脚本（`extract_signals.py`/`merge_interview_rules.py`/`validate_v2_all_blueprints.py`等），不是从零开发，是把它们Agent化。PTA这次把6个遗留扁平脚本迁移进skills/tools架构的经验，可以直接照抄。

### 5.1 分批策略：按风险递增，不要一次性搬完

PTA把6个脚本分成3批：批1最简单最独立的先做（PTA-DASH/EXT），批2其次（PTA-SCAN/DISCOVER，涉及切换哈希算法等真实改动），批3最复杂放最后（PTA-INTEL/INTEL-RW，两个脚本表面同构但数据模型完全不同，合并前要先判断能不能合并）。**每一批完成后都要跑一遍真实数据验证+全量测试，再进下一批**，不要囤积到最后一次性验证——批次之间互相独立，能更早发现问题。

### 5.2 迁移不是照搬：用新架构的原则筛一遍

迁移时至少要做三件事，而不是简单地把代码复制过去改个文件名：

1. **消除重复实现**：如果旧脚本自己维护了一套本可以复用现有工具的逻辑（比如自己实现文件哈希/diff，而框架里已经有通用的哈希/diff工具），迁移时换成复用现有工具，不要把重复实现也原样搬过去。PTA迁移SCAN时发现它自己维护一套哈希逻辑（且用的哈希算法跟项目其余部分不一致），换成了复用已有的通用diff工具。
2. **该删的功能就删，不要照单全收**：旧脚本里如果有跟新架构的原则冲突的功能（比如违反"不写入目标项目目录"隔离原则的写入路径，或者只在部分旧脚本里存在、明显是历史包袱的跨系统集成），迁移时应该判断后直接删除，不要因为"旧脚本里有"就无脑保留。PTA迁移时删除了SCAN的内部定时循环（违反隔离原则）和INTEL的跨Agent状态上报集成（写入范围超出了Agent自己的边界）。**删除是个决策，要在执行记录里写清楚删除的理由**，不是悄悄漏掉。
3. **两个原本独立的实现表面相似，不代表可以强行合并成一套**：如果发现两个旧脚本"看起来做的是同一件事"，先深入对比数据模型/核心逻辑，如果底层模型确实不同（PTA的INTEL通用版 vs INTEL-RW专用版，字段完全对不上），合并的应该是**入口层**（自动判断该用哪个后端），不是把两套数据模型强行拍扁成一套——后者是没必要冒的风险，且很可能引入新bug。

### 5.3 同一个bug在两个地方独立出现，才是抽共享模块的信号

不要为了"看起来更整洁"就提前把代码抽成共享模块——过早抽象反而增加理解成本。**真正该抽的信号是：同一条业务逻辑被两个模块独立维护了各自的一份拷贝，而且已经真实验证到，同一个bug在两边分别复现了一次**（说明"一个地方改了、另一个地方忘了改"这类风险是真实存在的，不是假设）。PTA在批3才抽了`tools/rw_conventions.py`，正是因为发现"Rw项目数据格式的判断逻辑"独立存在于两个skill里，且真的各自踩了一次同一个bug。批1、批2迁移时没有提前抽象，是正确的克制。

### 5.4 迁移完成 ≠ 迁移完成，务必对着真实数据再走一遍第5步

见"第5步·5.2节·第5条原则"——旧脚本可能是很久以前针对某个真实项目写的，但迁移进新架构后的第一次验证，必须重新对着（最好是同一个或类似的）真实项目数据跑一遍，不能只信任"旧脚本以前跑过、逻辑原样保留、所以现在也没问题"这个假设。PTA迁移完三批之后，第一次真实数据验证就抓出了5个新/旧代码都没被测出来过的真实bug。
