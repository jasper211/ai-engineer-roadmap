## 一、设计背景

### 1.1 为什么需要 Agent 体系

Phase 1 完成了多个自动化脚本（信号提取、规则合并、一致性校验），但存在以下问题：

| 问题 | 表现 | Agent 化解决 |
|------|------|-------------|
| 脚本分散 | 每个脚本独立运行，无统一调度 | 主 Agent 统一编排 |
| 触发被动 | 需要用户手动执行 | 子 Agent 主动监控 + 触发 |
| 产出孤立 | 信号基线、规则空白、看板更新各自为政 | 端到端自动同步 |
| 认知断层 | 每次新对话需重新理解上下文 | Agent 配置持久化上下文 |
| **（v1.1新增）方法论各自发明** | PTA/VNW/AIT各自摸索验证标准，没有共享标准，AI工具间调用规范也不统一 | 方法论转正Agent统一沉淀验证标准；Obsidian巡检Agent统一AI工具调用准则 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **L3 端到端** | 业务产出型主Agent（PTA/VNW/AIT）对应一个完整的 L3 流程，子 Agent 对应 L4/L5 活动 |
| **单一职责** | 主 Agent 不跨域，PTA 管项目、VNW 管价值节点、AIT 管转型咨询 |
| **可组合** | 子 Agent 可被多个主 Agent 复用（如 PTA-S04 文档同步器可被 VNW 调用） |
| **渐进实现** | 先实现 VNW（业务价值最直接），再 PTA（通用性最强），最后 AIT（战略价值）——**实际推进顺序已偏离原计划，PTA先跑到v1.2.0，见九、实现路线图** |
| **（v1.1新增）验证独立** | 所有Agent的"验证"环节用Agent完成，不用人工；验证标准统一由方法论转正Agent维护，不各自发明（详见D-20260709-001） |
| **（v1.1新增）面向复用设计** | 部分Agent未来要推向公司使用，不只是个人资产——配置和任务知识库需可插拔（参照PTA v1.2.0的`pta_tasks.json`外置模式），不能硬编码个人工作路径/习惯 |

## 三、主 Agent 1：项目任务协同 Agent (PTA)

### 3.1 定位

通用型项目执行引擎，适用于任何项目的任务全生命周期管理。**（v1.1现状）已迭代到v1.2.0，是本体系里唯一走完Agent搭建SOP全部6步的Agent，草稿态待Mark确认归档。**

### 3.2 L3 端到端流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ L3-PTA-01│ → │ L3-PTA-02│ → │ L3-PTA-03│ → │ L3-PTA-04│ → │ L3-PTA-05│
│ 任务解析 │    │ 执行编排 │    │ 进度监控 │    │ 产出同步 │    │ 归档复盘 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 3.3 子 Agent 设计（v1.1现状：全部已实现）

| 子 Agent | 职责 | 输入 | 输出 | 现状 |
|----------|------|------|------|------------|
| **PTA-S01 意图解析器** | 理解用户指令，识别任务类型、优先级、约束条件 | 自然语言指令 | 结构化任务包（JSON） | ✅已实现，v1.2.0任务ID识别正则已泛化（不再只认P#-##） |
| **PTA-S02 执行调度器** | 分解任务为可执行步骤，调度工具（Python/browser-use/Git） | 任务包 | 执行计划 + 步骤清单 | ✅已实现，v1.1.0修复跨节点执行82%→100% |
| **PTA-S03 进度追踪器** | 监控任务执行状态，处理阻塞/异常，向用户汇报 | 执行步骤状态 | 进度报告 + 风险预警 | ✅已实现 |
| **PTA-S04 文档同步器** | 任务完成后自动更新看板、执行记录、README、Git 提交 | 任务完成信号 | git commit + 看板更新 | ✅已实现，⚠️v1.1复核修正：实际不存在`--execute`参数，真实机制是`--dry-run`不传即真实执行（默认危险）；"不无人值守自动推送"这条安全属性成立，靠的是PTA-RUN主编排器根本没把S04接入自动串联链，不是S04自身有确认门槛，见Agent验证标准清单_v1.0.md |
| **PTA-S05 归档复盘器** | 生成执行记录、沉淀经验教训、更新 F3 教训库 | 任务完整历史 | 执行记录.md + 教训库条目 | ✅已实现 |
| **（v1.1.0新增）PTA-RUN 主编排器** | 自动串联S01→S02→S03→S05 + 状态记忆 | 用户指令 | 全流程自动执行 | ✅已实现，PTA从"5个需人工依次调用的脚本"变成真正符合SOP定义的Agent |

**v1.2.0关键改造（面向公司推广的第一个实证）**：任务知识库外置为可插拔JSON（`pta_common.py`+`pta_tasks.json`），此前S01/S02硬编码只认本项目9个任务，改造后`--project-root`下的项目自定义`pta_tasks.json`优先，兜底本项目内置默认值，**已用真实假外部项目验证+在真实的Rw权益项目上验证过**——这是本体系里第一个真正证明"可以推向公司/其他项目使用"的案例，其他Agent做类似改造时可以直接参考这个模式。

### 3.4 待办：按Agent搭建SOP v1.1第5.2节改造验证环节

PTA目前的`test_pta_integration.sh`是集成测试层（脚本判定通过/失败），但端到端测试这一层严格说还没有按D-20260709-001的四条原则系统性复核过——这是PTA走完6步之后，第一个要回头补的东西（详见D-20260709-001"验证方式"字段）。

业务型工作流引擎，专门处理价值节点 → 信号 → 规则空白 → SOP 的完整链路。**（v1.1现状）早期阶段，仅有config.json草稿，子Agent尚未系统性实现。**

### 4.2 L3 端到端流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ L3-VNW-01│ → │ L3-VNW-02│ → │ L3-VNW-03│ → │ L3-VNW-04│ → │ L3-VNW-05│
│ 节点解析 │    │ 信号提取 │    │ 基线合并 │    │ 规则空白 │    │ SOP底稿 │
│         │    │         │    │         │    │ 地图生成 │    │ 生成   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
     ↑                                                              ↓
     └──────────────────────────────────────────────────────────────┘
                         反馈闭环（清单更新后重新触发）
```

### 4.3 子 Agent 设计

| 子 Agent | 职责 | 输入 | 输出 | 对应现有脚本 |
|----------|------|------|------|------------|
| **VNW-S01 清单监控器** | 监控价值节点清单 Excel 变更（mtime/哈希） | 文件系统事件 | 变更通知 + 变更摘要 | 已有草稿脚本 |

| **VNW-S03 信号提取器** | 基于节点数据，按三层递进法提取 7 类信号 | 节点数据 + 蓝图 | 信号基线（Markdown） | Phase1的extract_signals.py（706行，已验证） |
| **VNW-S04 基线合并器** | 合并自动基线与手工基线（访谈规则继承） | auto_vX.md + manual_vY.md | 完整基线 auto_vZ.md | Phase1的merge_interview_rules.py（228行，已验证） |

## 六、主 Agent 4：方法论转正 Agent（v1.1新增）

### 6.1 定位

治理型Agent，不对应具体业务产出，负责监测其余四个Agent（含自己）的执行记录/踩坑记录，把可复用的规则回写进标准文档——是"设计-执行-核验-修正"闭环里"修正"这一环的专职承担者。**目前未启动，本次首次提出。**

### 6.2 为什么需要它（背景）

Agent搭建SOP第6步"文档归档"里已经要求生成"执行记录.md"（含踩坑记录），但此前没有机制把这些踩坑记录系统性地提炼成规则、回写进标准文档——PTA v1.1.0/v1.2.0的多次改造（任务知识库外置、验证环节改造）实际上都是"踩坑→手动回写SOP"的例子，只是靠Jasper自己盯着做，没有Agent化。

### 6.3 L3 端到端流程（草案，待走完SOP第1-2步细化）

```
┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────────┐
│ 执行记录监测 │ → │ 规则提炼   │ → │ 双向回写   │ → │ 验证标准    │
│           │    │           │    │(本项目SOP+ │    │ 清单维护   │
│           │    │           │    │ EA项目定义) │    │           │
└───────────┘    └───────────┘    └───────────┘    └───────────┘
```

### 6.4 职责边界

- **输入**：各Agent的执行记录.md、踩坑记录、集成测试失败日志
- **输出**：更新后的`Agent搭建SOP.md`（本项目）+ 更新后的`Skill与Agent_定义与功能说明.md`（EA项目，跨项目写入需明确权限边界，见D-20260709-001"承担风险"）+ 持续维护的"Agent验证标准清单"（供其他Agent的验证环节复用）
- **验证者是谁**：D-20260709-001已标注这是未决问题——方法论转正Agent自己产出的新规则要不要被验证、由谁验证，跟"验证一律用Agent不用人工"这条总原则可能有张力，需要Jasper后续拍板

### 6.5 触发场景

| 事件 | 方法论转正Agent响应 |
|------|---------|
| 某个Agent归档节点（SOP第6步）完成 | 读取执行记录 → 扫描踩坑记录 → 判断是否有可复用规则 |
| 某个Agent的验证环节命中新的异常特征 | 提炼该特征 → 追加进"Agent验证标准清单" |
| 累计3个Agent的踩坑记录后 | 触发一次SOP复盘更新（对应现有SOP"每完成3个Agent后复盘优化"的迭代规则） |
