## 一、双方定位（不重复设计，引用既有权威文档）

依据《三大主Agent体系架构_v1.3》§6：
- **OB**：个人Agent体系架构（五Agent：PTA/VNW/AIT/方法论转正Agent/OB）里的
  基础设施/治理层Agent，三条能力线（巡检/检索服务/概念笔记提炼），服务
  对象是其余四个业务Agent
- **方法论转正Agent**：两条并列线——**内部治理线**（监测PTA/VNW/AIT执行
  记录，提炼规则回写标准文档，本次不启动）+ **行业自学习线**（对外扫描
  行业实践反哺内部方法论，本次只做"AI协同方法论"方向，早期原型阶段）

**关键澄清（2026-07-23核实，纠正此前不准确的表述）**：OB现有实现**不是
纯RAG**，是结构化Wiki层（`concept_atom`笔记+`entity_ref`精确匹配枢纽+
`authority_layer`/`status`/`superseded_by`字段+entity_ref同组矛盾检测
看板）和向量层（Obsidian Smart Connections插件的embedding）并存的混合
系统。此前对话里"OB=RAG范式，方法论Agent=Karpathy compile范式"这个
二分法不准确，不再使用；真正的差异是**源材料的易变性**（OB处理的项目
文件持续更新迭代，需要版本去重/校准/退役这类质量刹车机制）和**整合
深度**（OB做原子级提炼，方法论转正Agent做Karpathy式的概念页级综合，
带跨文献合并和交叉引用维护）。

**已证伪、不应再引用的说法**：一份"元宝"(腾讯AI)对Karpathy方法论的转述
里，"`.meta/embeddings.json`钩子"和"个人/企业规模决定compile/RAG是否
融合"这两点，经回查[Karpathy原文](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
确认均不在原文中。设计依据以Karpathy原文+经核实可信的讲座转录稿
（`07-23_讲座：AI驱动的个人知识库构建`）为准，不引用元宝转述里的这两点。

## 二、Vault顶层目录归属（2026-07-23确认，推翻2026-07-15"三个顶层目录"旧决策）

vault（`OB知识库_vault/`）顶层按**知识产品的性质/来源**分区，不是按代码
仓库结构镜像。确认为**4个顶层目录**：

| 顶层目录 | 归属 | 内容性质 |
|---|---|---|
| `EA流程架构项目/` | OB | 从EA项目白名单文档提炼的项目知识原子 |
| `Jasper AI协同经验引擎/` | OB | 从"Jasper AI协同经验引擎"代码仓库白名单文档（三大主Agent体系架构、Mark_AI经验合集学习参考）提炼的Agent协同经验知识——**注意**：这个vault文件夹跟同名的代码仓库文件夹是两回事，代码仓库是"怎么搭建Agent"本身（按01-11骨架组织），这个vault文件夹是"从代码仓库特定文档提炼出的知识产品" |
| `方法论知识库/` | 方法论转正Agent | 对外采集的行业知识（AI协同方法论方向已启动，企业流程架构方向未启动），Karpathy三层结构（raw/wiki/CLAUDE.md） |
| `MOC/` | 共用 | 导航枢纽，两边都可能需要新增聚合页 |

**边界铁律**：谁的知识产品，谁负责往对应顶层目录写；不越界写对方目录；
OB的每日自动化白名单明确只扫`EA流程架构项目`和`Jasper AI协同经验引擎`
两个来源目录（见`提炼标准_v1.0.md`），不会碰`方法论知识库/`。

## 三、Schema兼容契约

两边各自的frontmatter字段**允许独立存在**（OB的`concept_atom`五+N个
字段 vs 方法论转正Agent的`source_url`/`collected_at`/`staleness_review_
date`/`info_type`/`evidence_basis`五个字段，语义不同，不能互相替代——
一手项目原子和二手行业信息的可信度/时效语义本来就不一样）。

**唯一的兼容要求**：若未来要复用OB现有的entity_ref枢纽/矛盾扫描机制，
方法论卡片需要额外挂载`entity_type`/`entity_ref`两个字段，**命名和
取值方式**需要跟OB的`concept_atom`schema一致。具体怎么填、什么时候
开始填，**双方约定等方法论知识库产出第一批真实卡片、拿真实数据验证过
一遍后再定**，不在设计阶段假想。

## 四、融合机制（四件套，设计已定，代码未动）

1. **硬关联**：方法论卡片若能对上EA正式编码（entity_ref），走现有
   entity_ref枢纽机制，同编码下项目原子和方法论卡片混合挂载
2. **软关联**：语义相似内容走embedding聚类（复用"非正式主题"枢纽
   threshold=0.72+LLM连贯性判断），方法论卡片写法可能跟项目原子差异大，
   **小样本先验证聚类阈值是否仍准确，再全量上**
3. **晋升机制**：项目原子→方法论卡片的候选筛选，复用现有字段不发明
   新指标：`confidence=HIGH` 且 `status=生效` 且 `backlink数量>=3`
   （2026-07-22在EA项目764个原子上实测，产出11个候选，规模合理）且
   未被entity_ref矛盾扫描标记冲突。候选清单由OB一次性查询脚本产出，
   "怎么合并进方法论卡片"归方法论转正Agent负责，OB不越界代做。
   **本次交接澄清**：晋升机制对"行业信息"和"项目文档"两个方向都适用，
   不是单向的项目→方法论——行业知识反过来给项目原子当校验标尺，见下条
4. **冲突检测（双向）**：同entity_ref下若既有项目原子又有方法论卡片，
   拿方法论卡片当标尺比对项目原子（"项目做法 vs 行业权威说法"是否冲突），
   扩展现有entity_ref矛盾扫描逻辑，机制复用，不另起一套

**代码层面现状（诚实披露）**：`cluster_atoms.py`的`AtomClusterer`初始化
只传一个`project_name`，一次只扫一个项目文件夹——即使schema完全兼容，
枢纽构建代码现在也不会自动跨文件夹把方法论卡片和项目原子拉进同一个
entity_ref hub。**双方已达成一致**：entity_ref精确匹配这部分改成跨
文件夹扫描风险低，可以先改；embedding聚类部分因为差异风险要先小样本
验证。**这个代码改动记为明确的下一步，不是现在**——等方法论知识库
产出第一批真实卡片、双方拿真实数据验证过效果之后再改，不对着假想场景
先写自动化代码（这是这次交接反复强调的原则，不是新原则）。

## 五、检索归属（澄清，非最终结论）

OB现有两层检索能力（结构化entity_ref/hub层 + Smart Connections向量层）
并存。行业原子（方法论转正Agent的产出）该优先走哪一层、两层怎么分工，
**尚未正式确认**，不预设结论——这不是"要不要接入向量检索"的问题（向量
层本来就已经存在），是"新知识该怎么被这两层现有能力覆盖"的问题。
