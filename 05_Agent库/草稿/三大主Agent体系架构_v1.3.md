## 一、设计背景

### 1.1 为什么需要 Agent 体系

Phase 1 完成了多个自动化脚本（信号提取、规则合并、一致性校验），但存在以下问题：

| 问题 | 表现 | Agent 化解决 |
|------|------|-------------|
| 脚本分散 | 每个脚本独立运行，无统一调度 | 主 Agent 统一编排 |
| 触发被动 | 需要用户手动执行 | 子 Agent 主动监控 + 触发 |
| 产出孤立 | 信号基线、规则空白、看板更新各自为政 | 端到端自动同步 |
| 认知断层 | 每次新对话需重新理解上下文 | Agent 配置持久化上下文 |
| 方法论各自发明 | PTA/VNW/AIT各自摸索验证标准，没有共享标准 | 方法论转正Agent统一沉淀验证标准 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **L3 端到端** | 业务产出型主Agent（PTA/VNW/AIT）对应一个完整的 L3 流程，子 Agent 对应 L4/L5 活动 |
| **单一职责** | 主 Agent 不跨域，PTA 管项目、VNW 管价值节点、AIT 管转型咨询 |
| **可组合** | 子 Agent 可被多个主 Agent 复用 |
| **渐进实现** | 实际推进顺序已偏离原计划（PTA先跑到v2.8.2），**v1.3起进一步偏离"严格排队"假设**：VNW与AIT可以同时启动，见九节 |
| **验证独立** | 所有Agent的"验证"环节用Agent完成，不用人工；验证标准统一由方法论转正Agent维护 |
| **面向复用设计** | 部分Agent未来要推向公司使用，配置和任务知识库需可插拔 |
| **（v1.3新增）真实案例先于抽象框架** | AIT的"SOP三轨道"最初是纯概念推导，用L3-COM真实规划材料校准后发现真实实践更精细——抽象设计要拿真实案例反复验证，不能只在概念层打转 |

## 四、主 Agent 2：价值节点驱动工作流 Agent (VNW)

### 4.1 定位——不只是流程处理引擎，是"未知转已知"的发现引擎

业务型工作流引擎，专门处理价值节点 → 信号 → 规则空白 → SOP 的完整链路。**（v1.3重新定义核心价值）**VNW的意义不是单纯推进"已经识别的节点"，而是把"不管我们现在知不知道"的盲区都逼出来变成"知道"——面板/评估不能只看已识别节点的完工率，还要体现"全域应覆盖但还没触达"的盲区提示，否则会掩盖没被发现的空白。**（现状）**早期阶段，仅有config.json草稿，子Agent尚未系统性实现。

### 4.2 节点锚点模型（v1.3新增）

VNW的最小追踪单元是**价值节点**，一个节点可能对应**多个L4交付物**。例如`L3-COM 佣金全链路管理`这个节点，对应主链10个L4（COM-01~COM-15）+ 支链8个L4，共18条——节点是锚点，L4是节点下的具体活动分解，规则挂在L4之下，不是挂在节点这个粗颗粒度上。

### 4.3 规则级溯源模型（v1.3新增）

VNW的溯源逻辑跟EA项目的溯源精神一致（都是为了"输出的东西可以在过程中溯源"），但**VNW不执行EA项目那套01→02→03分层晋升锁定机制**。VNW要体现的是：

- 每一条规则要标注**在SOP的哪个段落**（如"§2.3 异常触发条件"）
- 每一条规则要表述**源头来自哪**——机器规则/人的规则的源头通常是01层访谈材料，但**协同新规则**（试运行中冒出来的新规则）的源头往往是执行记录/试运行日志本身，不是传统01层材料，这点需要单独适配
- 这套对应关系**直接维护在02层过程文档内**，边产出边标注，不等交付晋升那一刻才补标注——因为VNW产出物大部分时候就停留在02层工作，不像EA走严格的03层锁定发布

| 子 Agent | 职责 | 输入 | 输出 | 对应现有脚本 |
|----------|------|------|------|------------|

## 五、主 Agent 3：AI 协同转型 Agent (AIT)

### 5.1 定位——不是独立轨道，是VNW流水线在SOP之后的延伸段

**（v1.3重新定义，替代v1.2"战略型咨询引擎"的独立定位）**VNW产出的SOP本质上就是"AI的SOP"——AIT不是一条独立于VNW、需要自己从头识别需求的战略咨询轨道，**AIT的任务来源就是VNW分析出的SOP本身**：VNW某个节点一旦被分析、产出SOP，就自然触发AIT介入，把这份SOP正式构思成Agent搭建的任务面板。

因为VNW是全公司范围推动（周期长、体量大），**不必等VNW全部节点走完，可以拿"已经推动到SOP阶段的节点子集"作为MVP，AIT与VNW同时启动**，而不是排队等待——两者是同一条流水线的前后段。**（现状）**早期阶段，仅有config.json+需求定义.md，但已有一份完整的真实手工案例可供校准（见5.4节）。

### 5.2 SOP 的三轨道拆解

一份SOP不是铁板一块，AIT把SOP内部的规则按三类拆成三条不同性质的处理轨道：

| SOP里的规则类型 | AIT该怎么处理 | 产出形态 |
|---|---|---|
| **机器的规则**（确定性逻辑） | 标准五步走：需求识别→流程分析→AI化设计→Agent构建→SOP交付 | 可执行的Agent/Skill |
| **人的规则**（需要人工判断/裁量，或涉及不可逆风险的强制关卡） | 不是"全自动化"，产出物是"人机边界设计"——哪里必须停下来问人 | 介入点设计文档，不是脚本 |
| **协同新规则**（试运行中冒出来的，还不稳定） | 不直接进build流程，先进"观察/验证"暂存区 | 待验证条目，稳定后转正为前两类之一 |

### 5.3 协同新规则的判定机制

判定"协同新规则"是否稳定、可以转正，基于三路输入做**多方评估**，最终**人工判断**拍板，不是纯自动化判定：

1. **VNW输出的SOP**——这条新规则跟SOP里已有规则是否冲突/重复
2. **之前理想化的agent设计**——这条新规则是否已经在原始设计构想里被预见过
3. **skill自动化评估**——即5.4节校准出的四维就绪度矩阵/Tier打分等自动化评分工具

**评估结果不只是"稳定/不稳定"的二元判定**，还要反馈进每个Agent自己的规划与搭建过程——例如5.4节L3-COM案例里发现的"技术分数不能替代资金风险判断"这条教训，后续应该反过来修正评分框架本身，而不只是给单个节点打个分就结束。

不稳定、暂时无法转正的协同新规则，流入方法论转正Agent的内部治理线（见六节）。

### 5.4 真实案例校准：L3-COM 佣金全链路管理Agent规划

**（v1.3新增小节）** `02_过程成果-工作产出/规则分析（Jasper）/Agent规划与搭建/L3-COM_佣金全链路管理Agent/` 下已有8份文档（规划分析/卡点与前置条件/分期搭建计划/佣金数据表全景/流程蓝图重塑方案/四维就绪度矩阵/ABCD看板/原生Agent运行原理确认），是目前唯一一个完整走过"SOP→AI化设计→分期搭建"全过程的真实案例——虽然是Jasper手工分析产出，不是AIT这个Agent自动跑出来的，但正好可以拿来反推AIT真正建成后该长什么样。校准出的关键发现：

- **四维就绪度矩阵（R/D/P/O）是"多方评估"的真实雏形**：每个L4从规则(R)/数据(D)/流程(P)/组织(O)四个维度各打0-5分，总分≥14=可施工、8-13=待补齐、<8=暂搁置。这就是5.3节"skill自动化评估"要落地的具体形态——目前是Jasper人工评分，AIT真正建成后这一步应该自动化。
- **Tier分类（Auto/Aug/Hybrid/Human gate）比抽象的"机器规则/人的规则"二分更精细**：尤其"Human gate"这个类别，不是由规则本身的模糊性决定的，而是由一条独立的风险覆盖原则决定的（见下）。
- **关键教训——风险覆盖原则**：L3-COM案例发现"D1-D6纯技术评分不包含资金/合规风险敞口这个维度——只要涉及真实资金转出/账户注销这类不可逆资金动作，无论技术分数多高，默认保留Human gate，除非业务方明确书面豁免"。这条本身就是**方法论转正Agent内部治理线的候选第一批种子案例**（见六节）。
- **验证可以是三级串联检查链，不是单点**：L3-COM的验证设计是"COM-11自动对账→COM-13人工强制关卡(资金安全)→COM-17合规独立评估(角色与COM-13不同)"，说明5.2节"人的规则"轨道内部还可以细分出"谁来审"这一层，AIT的设计框架需要支持多角色审批流。

### 5.5 触发场景

| 事件 | AIT 响应 |
|------|---------|
| VNW某节点SOP产出完成 | 拆解为三轨道 → 机器规则走五步构建，人的规则设计介入点，协同新规则进观察区 |
| "财务部门想引入 AI 辅助对账" | 对照VNW是否已有对应节点SOP，有则直接进三轨道拆解，无则先反馈VNW去分析 |
| 协同新规则观察期判定稳定 | 转正为机器规则或人的规则，回填SOP新段落 |
| 协同新规则观察期判定不稳定 | 流入方法论转正Agent内部治理线 |

### 7.1 定位

**v1.2（07-15）曾写"OB还在等待接手指令、PTA↔OB两边都不动"，这个门槛事实上已经被跨过，v1.2的§7.4/§9.4描述已过时**。截至2026-07-18（OB v0.4.5）的真实进展：

### 7.4 写入触发的理想设计：OB自巡检

**范围声明（重要）**：**OB自身的搭建/优化工作在另一个独立项目里进行，不在本项目（AI工程能力整改项目）的范围内**。本节只记录PTA/VNW/AIT与OB对接的接口设计意图，不代表本项目会去实现OB的写入侧或自巡检机制——那是另一个项目的工作，这里只是提前对齐方向，避免后续对接时口径不一致。
