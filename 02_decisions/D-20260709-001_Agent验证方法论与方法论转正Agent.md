## 决策条目

| 字段 | 内容 |
|------|------|
| **决策编号** | D-20260709-001 |
| **决策时间** | 2026-07-09 |
| **决策场景** | 个人Agent体系（PTA/VNW/AIT/OB巡检）规划系统化过程中，明确"验证"环节的架构要求，同时补一个之前没有归属的职能空格子（方法论沉淀） |
| **决策问题一** | Agent五要素里的"验证"，用人工做还是用Agent做？如果用Agent做，靠什么保证不是自己给自己打好评？ |
| **决策问题二** | "方法论转正"（发现系统性问题→固化成规则→回写进标准文档）这个反复出现的职能，要不要单独建一个Agent？ |
| **最终选择一** | 验证环节一律用Agent，不引入人工验证；配套构建"Agent验证方法论"，用四条原则保证独立性，不是找另一个LLM"读一遍觉得对不对" |
| **最终选择二** | 新增"方法论转正Agent"作为第4个主Agent，与PTA/VNW/AIT并列 |
| **Agent验证方法论·四条原则** | ①**确定性检查优先**：能用脚本/规则判定对错的，不用LLM主观判断——PTA现有`test_pta_integration.sh`已经是这个原则的实例（shell脚本判定通过/失败，不是找AI看）；②**独立数据源交叉核验**：评估者的判断依据不能只看生成器自己产出的东西，要有独立于生成过程之外的锚点（例：S04文档同步器的验证不是"AI读一遍commit信息觉得写得好"，是核对"是否真的push成功、文件是否真的变更"）；③**异常特征清单优先于全量通读**：不让评估者逐字读全部输出再给感想，是维护一份具体、可检测的异常模式清单（如"分数区间和结论对不上""模板化复制粘贴"这类EA项目里已验证过的具体异常特征），评估者专门扫这些特征；④**验证标准独立文档化、跨Agent复用**：不能PTA和VNW的验证标准各自发明，要有一份共享的"Agent验证标准清单"，由方法论转正Agent维护 |
| **方法论转正Agent·职责边界** | 监测各Agent的执行记录/踩坑记录（Agent搭建SOP第6步已经在生成这些"执行记录.md"）→提炼可复用规则→回写进两处：本项目内`Agent搭建SOP.md`本身，以及EA项目那边`Skill与Agent_定义与功能说明.md`（两边共用同一套"Agent验证标准"时，需要同步更新，不能只改一边） |
| **未决问题（需要后续自己判断，本次不替你定）** | 方法论转正Agent自己产出的"新规则"要不要被验证、由谁验证——EA项目里"资金安全强制关卡"的逻辑是"规则本身要不要改"这类判断比"规则执行结果对不对"责任更重，可能值得单独保留人工确认这一层；但这跟"验证一律用Agent"这条总原则有张力，两者哪个优先，留给你自己拍板，不在本次决策范围内 |
| **承担风险** | ①"确定性检查优先"意味着有些真正模糊、需要主观判断的验证点，可能被勉强套成"确定性检查"，反而降低了判断质量——需要在后续验证方法论迭代时警惕；②方法论转正Agent同时维护两个项目的文档（本项目+EA项目），存在权限/路径管理复杂度，需要在实现时明确它对EA项目文件是只读引用还是有写权限 |
| **验证方式** | 四条原则先用于PTA现有`test_pta_integration.sh`的复核——检查其8项集成测试是否已经符合这四条原则，不符合的地方作为方法论转正Agent的第一个真实案例 |

## 验证结果（2026-07-09，PTA复核）

四条原则套到PTA现有`test_pta_integration.sh`（8个用例，复核前全部通过）上，发现3类问题，均已修复并重新跑通（改后9个用例全部通过，exit 0）：

1. `set -e`没配`pipefail`——`python3 ... | tail -N`模式下python崩溃会被tail的退出码掩盖，此前靠后续文件存在性检查侥幸兜底，不是设计保证 → 已加`pipefail`，并把两处`grep -q`后置查`$?`的写法改成`if pipeline; then`直接判断（避免pipefail下set -e提前终止）
2. **架构文档里"真实git push需要显式`--execute`确认"的说法是错的**——代码里根本没有`--execute`参数，真实机制是`--dry-run`不传即真实执行（默认危险，不是要额外加参数才危险）；"不会无人值守自动推送"这条安全属性本身成立，但成立的原因是PTA-RUN主编排器根本没把S04接入自动串联链，不是S04自身有确认门槛 → 已修正三大主Agent体系架构_v1.1.md对应描述
3. 8个用例全是正向验证，没有反向/边界用例测"不该发生的事有没有发生" → 新增Test 9，在隔离无remote的临时仓库里验证S04默认行为，安全（push必然本地失败，不碰真实远端），实测确认"不传--dry-run会真实commit"

**尚未覆盖**：原则3（异常特征清单）本次未真正建立，跨Agent通用的异常模式清单还没有沉淀；S03/S05等几处产出内容本身的校验仍薄弱，只做了存在性检查。这两项列入`Agent验证标准清单_v1.0.md`的待办。

## 关键经验

1. 延续D-20260702-002已经验证过的"确定性脚本+不确定性AI组合"原则——这次是把它从"工具选型"层面延伸到"验证"层面，不是新发明一套逻辑
2. "验证用Agent不用人工"和EA项目里"资金安全类必须强制人工关卡"这两条原则，本质上都是"生成器/评估器分离"的具体应用，只是后者多加了一条"某些类别的判断风险太高，独立评估者本身也得是人"的例外规则——方法论转正Agent的自我验证问题，可能正好落在这条例外的边界上
3. **（v1验证新增）"测试全过"不等于"测试设计没问题"**——PTA的8个用例复核前全部通过，但复核后发现的3类问题没有一条是"测试报错了"暴露出来的，全部是重新审视测试设计本身才发现的。这印证了原则2（独立数据源交叉核验）的必要性：验证者不能只看"生成者的测试是否通过"，要独立核实"这些测试本身覆盖到位没有"
4. **（v1验证新增）文档和代码不一致时，代码是准的**——架构文档里"--execute确认"这条描述是错的，本次是靠直接读`PTA-S04_文档同步器.py`源码才发现，不是靠读README/架构文档。以后任何"安全机制是什么"类描述，写文档前先去核代码，不能凭记忆或旧版本行为写
