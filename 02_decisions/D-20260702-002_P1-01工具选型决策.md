| **决策问题** | 第一个 Computer Use / Browser Use 任务应该用什么工具实现？ |
| **可选方案** | 纯 browser-use MCP / Claude Computer Use API / Python + browser-use 混合 |
| **方案A：纯 browser-use MCP** | 用浏览器打开 Markdown 和 Excel 网页版，通过 DOM 提取和 OCR 获取数据，再比对。优点：全程可视化，截图证据自然；缺点：Excel 解析不稳定，Markdown 表格提取依赖渲染，开发成本高 |
| **方案B：Claude Computer Use API** | 调用 Anthropic 官方 Computer Use API，让模型直接控制浏览器/桌面完成提取和比对。优点：模型能力强，适合复杂交互；缺点：成本高，对结构化数据任务没必要，响应慢 |
| **方案C：Python 脚本 + browser-use MCP 混合** | Python 负责解析 Markdown 和 Excel 的结构化数据，browser-use 负责打开生成的 HTML 报告并截图。优点：结构化解析稳定可靠，browser-use 只做它擅长的可视化验证，成本低，速度快 |
| **最终选择** | 方案C：Python 脚本 + browser-use MCP 混合 |
| **选择理由** | ① 价值节点比对是结构化数据任务，Python+pandas+正则最可靠 ② browser-use 的核心价值是「可视化验证」和「截图证据」，不是「数据解析」 ③ 混合方案成本最低、速度最快、最符合生产环境自动化习惯 ④ 为后续 Skill 化留下清晰边界：Python 脚本作为核心逻辑，browser-use 作为验证层 |
| **承担风险** | ① 需要维护两套工具（Python + browser-use），初期复杂度略高 ② 如果未来需要「模拟人类操作 Excel」而非「读取 Excel」，混合方案需要调整 ③ browser-use MCP 的本地服务器稳定性依赖环境 |

## 关键经验

1. **工具选择应基于任务特征**：结构化数据任务用脚本，可视化验证任务用 browser，不要反过来。
2. **browser-use 的定位**：它是「验证层」和「证据层」，不是「数据解析层」。
3. **混合方案是工程化常态**：生产环境中的 AI 自动化很少是单一工具，通常是「确定性脚本 + 不确定性 AI」的组合。
