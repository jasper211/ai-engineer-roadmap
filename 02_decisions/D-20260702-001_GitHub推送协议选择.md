## 决策条目

| 字段 | 内容 |
|------|------|
| **决策编号** | D-20260702-001 |
| **决策时间** | 2026-07-02 |
| **决策场景** | Phase 0 · GitHub 仓库初始化与首次 push |
| **决策问题** | 首次 push 到 GitHub 时遭遇 `LibreSSL SSL_connect: SSL_ERROR_SYSCALL`，应选择哪种协议/方式解决？ |
| **可选方案** | HTTPS + PAT / SSH 密钥 / GitHub Desktop / 切换 OpenSSL 后端 |
| **方案A：HTTPS + PAT** | 按 GitHub 官方推荐，用 Personal Access Token 作为密码 push。优点：官方默认流程，配置简单；缺点：macOS LibreSSL 与 GitHub TLS 握手失败，无法直接使用 |
| **方案B：SSH 密钥** | 生成 ed25519 密钥对，公钥添加到 GitHub，本地用 `git@github.com` 地址 push。优点：绕开 HTTPS/SSL 兼容问题，长期稳定，无需反复输入 token；缺点：初次配置步骤较多 |
| **方案C：切换 OpenSSL 后端** | 执行 `git config --global http.sslBackend openssl`。优点：可能保留 HTTPS 流程；缺点：只是绕过 LibreSSL，未根本解决网络层不稳定问题 |
| **方案D：GitHub Desktop** | 用图形客户端完成首次 push。优点：无需命令行；缺点：不符合 AI 工程师以 CLI 为主的工作习惯，且无法沉淀可复用知识 |
| **最终选择** | 方案B：SSH 密钥认证 |
| **选择理由** | ① 根本性解决 macOS LibreSSL 与 GitHub HTTPS 的握手问题 ② SSH 是工程师协作的标准方式，符合目标岗位要求 ③ 一次配置长期使用，比 PAT 更适合后续频繁提交 ④ 能同时积累 SSH 密钥管理经验 |
| **承担风险** | ① 私钥泄露风险（已设置空 passphrase，需保护好 `.ssh/id_ed25519`） ② 换电脑时需要迁移密钥或重新生成 ③ 企业防火墙可能拦截 22 端口（GitHub 支持 443 端口 SSH 作为 fallback） |
| **验证方式** | ① 成功执行 `git push -u origin main` 并看到 `branch 'main' set up to track 'origin/main'` ② GitHub 仓库页面正常显示 README 内容 ③ 后续新增 commit 能正常 push |

## 关键命令备忘

```bash
# 生成密钥
ssh-keygen -t ed25519 -C "jasper211@users.noreply.github.com"

# 查看公钥
cat ~/.ssh/id_ed25519.pub

# 设置远程地址为 SSH
git remote set-url origin git@github.com:jasper211/ai-engineer-roadmap.git

# 验证连接
ssh -T git@github.com
```

## 复盘要点

1. ** LibreSSL 不是 OpenSSL**：macOS 系统自带的 LibreSSL 与部分网络环境/服务端 TLS 配置存在兼容性问题，遇到 `SSL_ERROR_SYSCALL` 时优先尝试 SSH。
2. **一次性投入 vs 反复成本**：SSH 初始配置比 HTTPS+PAT 多几步，但后续每次 push 都更顺畅，长期 ROI 更高。
3. **知识资产化**：把一次排错过程写成决策日志，下次换环境或教别人时可以复用，而不是重新搜索。
