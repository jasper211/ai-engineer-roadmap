# 决策日志模板

> 每个关键决策都必须记录，避免"做了就忘"和"下次重来"。

---

## 📋 文档定位说明

| 字段 | 内容 |
|------|------|
| **文档定位** | 项目中的「决策档案馆」 |
| **核心作用** | ① 把隐性的判断过程显性化，防止"只记得结论、忘了理由" ② 让未来的自己或协作者理解当时为什么这样选 ③ 为复盘和迭代提供可追溯的依据 |
| **使用场景** | ① 做技术选型时 ② 选择部署平台时 ③ 决定优先级调整时 ④ 任何"如果半年后回看，可能想不起来为什么"的选择 |
| **维护责任** | Jasper 主责，每做一个关键决策就记录一条 |
| **迭代规则** | ① 决策被验证后更新状态为「已验证」 ② 决策被推翻后更新状态为「已推翻」并说明原因 ③ 模板本身不需要频繁迭代 |
| **关联文件** | [任务执行模板](./任务执行模板.md) · [周报复盘模板](./周报复盘模板.md) · [能力整改看板](../能力整改看板.md) |

---

## 决策条目

### 决策编号：D-{YYYYMMDD}-{NNN}

| 字段 | 内容 |
|------|------|
| **决策时间** | YYYY-MM-DD |
| **决策场景** | 例如：选择 Computer Use 技术方案 / 部署平台选择 / 模型路由策略 |
| **决策问题** | 需要解决什么问题？ |
| **可选方案** | 列出2-3个可行方案 |
| **方案A** | 描述 + 优缺点 |
| **方案B** | 描述 + 优缺点 |
| **方案C** | 描述 + 优缺点 |
| **最终选择** | 选择哪个方案？ |
| **选择理由** | 为什么选它？关键判断依据是什么？ |
| **承担风险** | 这个选择可能带来什么风险？ |
| **验证方式** | 如何验证这个决策是否正确？ |
| **决策人** | Jasper |
| **相关文件** | 链接到执行记录或产出物 |
| **状态** | 已验证 / 待验证 / 已推翻 |

---

## 示例

### D-20260630-001

| 字段 | 内容 |
|------|------|
| **决策时间** | 2026-06-30 |
| **决策场景** | Computer Use 技术选型 |
| **决策问题** | 第一个 Computer Use 任务用什么技术实现？ |
| **可选方案** | Claude Computer Use API / browser-use / Playwright+AI |
| **方案A** | Claude Computer Use API：官方能力，但成本高 |
| **方案B** | browser-use：开源，成本低，社区活跃 |
| **方案C** | Playwright+AI：灵活，但需要自己搭建推理层 |
| **最终选择** | 先用 Claude Computer Use API 跑通一个场景，再用 browser-use 复现同一任务做对比 |
| **选择理由** | 需要同时积累"前沿 API 实操"和"开源方案对比"两种经验 |
| **承担风险** | Claude API 成本较高，需控制测试规模 |
| **验证方式** | 两个方案都产出 demo，对比成功率、成本、稳定性 |
| **决策人** | Jasper |
| **相关文件** | 01_execution/P0-01_ComputerUse/ |
| **状态** | 待验证 |
